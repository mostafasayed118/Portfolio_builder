## 1. Project Purpose

Portfolio-Fixer is a full-stack portfolio CMS. The public portfolio SPA (`artifacts/portfolio`) showcases Mustafa's skills, projects, experience, and certifications to recruiters. The admin dashboard (`artifacts/admin`) is a Clerk-authenticated CMS for managing all portfolio content. The Express 5 API server (`artifacts/api-server`) handles CV generation, image uploads, contact form ingestion, and admin CRUD operations. Supabase (PostgreSQL) is the database and storage backend.

## 6. Auth Model

### Clerk JWT Verification (`api-server/src/middleware/adminAuth.ts`)

The `adminAuth` middleware supports three auth methods:

1. **API Key** (`x-admin-key` header) — timing-safe comparison against `ADMIN_API_KEY` env var. Works without Clerk. Creates/looks up a default admin user in `users` table.

2. **Clerk JWT** (`Authorization: Bearer <token>`) — uses `@clerk/backend`'s `verifyToken()` with `CLERK_SECRET_KEY`. Looks up user by `clerk_id` or email in `users` table. Auto-provisions new users.

3. **Reject** — if neither method succeeds, returns 401.

**Email allowlist:** `VITE_ADMIN_EMAILS` env var (comma-separated). Clerk JWT is only accepted if the token's email is in this list.

### Supabase RLS

- **Public tables** (hero_content, skills, projects, etc.): `FOR SELECT USING (true)` — anyone can read. `FOR ALL USING (is_admin())` — only admin can write.
- **Messages table**: public INSERT (contact form), admin-only SELECT/UPDATE/DELETE.
- **Admin uses service role key** (bypasses all RLS).

### Portfolio Auth

No authentication. Portfolio uses Supabase anon key with public-read RLS policies.

### Admin SPA Auth

Clerk React SDK (`@clerk/clerk-react`). `<ProtectedRoute>` component checks `useAdminAuth` hook against `APP_ADMIN_EMAILS`.

## 11. Known Gotchas

- **`admin/src/lib/convex.ts`** — named after old Convex backend but is actually a Supabase client wrapper. Historical naming artifact.
- **`lib/api-client-react/`** — dead code directory. Generated by orval from OpenAPI spec but never imported by any artifact. Can be deleted.
- **`lib/db/src/contactMessages.ts`** exists alongside `messages.ts` — `contactMessages.ts` may be an older version; `messages.ts` is the active module.
- **Replit plugins** in catalog (`@replit/vite-plugin-*`) — dev-only, won't work outside Replit. Not consumed in production builds.
- **CSP allows `'unsafe-inline'`** for styleSrc — required for TailwindCSS dynamic styles. scriptSrc is `'self'` only.
- **Admin uses service role key** in the browser (`VITE_SUPABASE_SERVICE_ROLE_KEY`) — this key bypasses RLS and is exposed to the client bundle. Intentional for admin-only deployment but must be kept secret.
- **`isProduction` in app.ts** is referenced at line 17 (inside `isValidUrl`) before being declared at line 68. Works due to hoisting of `const` in module scope at runtime, but is a code smell.
- **Migration numbers have gaps** — 010, 016–019 were removed or consolidated. This is cosmetic but confusing.
- **Portfolio has static fallback** — when `VITE_SUPABASE_URL` is not set, portfolio renders from `src/data/*.ts` files. This is intentional for development without Supabase.
- **`messages` table name** — types.ts exports as `Message` but the DB table is `messages`. The contact route inserts into `"messages"`. No `contact_messages` table exists despite docs mentioning it.
