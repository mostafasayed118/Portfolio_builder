### Visual consistency

#### [UX-001] Admin card content padding still inconsistent
- **Severity:** Low
- **App:** admin
- **Location:** `artifacts/admin/src/pages/ProjectsManager.tsx:137` (`pt-4 pb-4`) vs `artifacts/admin/src/pages/CvManager.tsx:173` (`pt-5 pb-4`)
- **Observed:** Project cards use 16px top padding while CV info cards use 20px. The pattern varies across manager pages with no documented rationale.
- **Expected:** Consistent card content padding across all admin list views.
- **Impact:** Subtle visual inconsistency when switching between manager pages.
- **Suggested fix:** Standardize all card content to `pt-5 pb-4` or `py-4`.

#### [UX-002] SectionLabel component used inconsistently
- **Severity:** Low
- **App:** portfolio
- **Location:** `artifacts/portfolio/src/components/AboutSection.tsx:124` uses `<SectionLabel>` while `artifacts/portfolio/src/components/SkillsSection.tsx:196`, `ProjectsSection.tsx:100`, `ExperienceSection.tsx:79`, `CertificationsSection.tsx:225`, `ContactSection.tsx:174` use inline `<div>` with identical markup
- **Observed:** The `SectionLabel` component exists and is used in AboutSection, but all other sections duplicate its markup inline (`inline-flex items-center gap-2 text-xs font-semibold tracking-widest uppercase text-primary bg-primary/10 border border-primary/20 px-3 py-1.5 rounded-full mb-4`).
- **Expected:** All sections should use the shared `SectionLabel` component.
- **Impact:** If the section label design needs to change, 5 sections must be updated manually instead of one component.
- **Suggested fix:** Replace inline markup in all 5 sections with `<SectionLabel>{t.section.title}</SectionLabel>`.

#### [UX-003] Toast variant colors hardcoded in shared UI library
- **Severity:** Medium
- **App:** both
- **Location:** `lib/ui/src/components/primitives/toast.tsx:33-35` — `border-green-500/50 bg-green-500/10`, `border-yellow-500/50 bg-yellow-500/10`, `border-blue-500/50 bg-blue-500/10`
- **Observed:** The shared toast component uses hardcoded Tailwind color classes for success, warning, and info variants instead of semantic design tokens. The destructive variant at line 81 uses `text-red-300`, `focus:ring-red-400`, `focus:ring-offset-red-600`.
- **Expected:** Toast variants should use semantic tokens (`--success`, `--warning`, `--info`) or shadcn's built-in variant system.
- **Impact:** Toast colors cannot be themed consistently with the rest of the design system; dark mode colors may not adapt properly.
- **Suggested fix:** Replace hardcoded color classes with CSS variable references or use the design system's semantic color tokens.

#### [UX-004] Admin Header toggle buttons below 44px

#### [UX-005] Admin manager edit/delete buttons still at 32×32px

### Layout & structure

#### [UX-009] Suspense fallback is completely blank
- **Severity:** Critical
- **App:** portfolio
- **Location:** `artifacts/portfolio/src/App.tsx:19` — `<Suspense fallback={<div className="min-h-screen" />}>`
- **Observed:** When lazy-loaded sections are being fetched, the user sees a completely blank white screen with no loading indicator, spinner, or skeleton. This can last several seconds on slow connections.

### UX patterns & usability

#### [UX-018] Manager forms have no client-side validation
- **Severity:** Critical
- **App:** admin
- **Location:** `artifacts/admin/src/pages/ProjectsManager.tsx:65-83`, `ExperienceManager.tsx:57-70`, `CertificationsManager.tsx:52-79`
- **Observed:** The save handlers for Projects, Experience, and Certifications managers have no client-side validation. Users can save entries with empty titles, missing required fields, or invalid URLs. The only validation is the server-side response, which returns a generic error after the network round-trip.
- **Expected:** Forms should validate required fields before attempting to save.
- **Impact:** Users waste time filling out forms only to discover errors after a network request fails. Empty or invalid entries may be saved to the database.
- **Suggested fix:** Add client-side validation: `if (!editing.title?.trim()) { toast({ title: "Title is required", variant: "destructive" }); return; }`

#### [UX-019] CvManager has no file size validation
- **Severity:** Medium
- **App:** admin
- **Location:** `artifacts/admin/src/pages/CvManager.tsx:47-94`
- **Observed:** The CV upload only checks for PDF file type (`file.type.includes("pdf")`) but does not validate file size. Large PDF files (e.g., 50MB+) will attempt to upload, potentially timing out or consuming excessive bandwidth.
- **Expected:** File uploads should have a maximum size limit with clear user feedback.
- **Impact:** Users may attempt to upload very large files, resulting in slow uploads, timeouts, or storage quota issues.
- **Suggested fix:** Add size check: `if (file.size > 10 * 1024 * 1024) { toast({ title: "File too large", description: "Maximum file size is 10MB", variant: "destructive" }); return; }`

#### [UX-020] SeoManager has no URL format validation
- **Severity:** Medium
- **App:** admin
- **Location:** `artifacts/admin/src/pages/SeoManager.tsx:72-94`
- **Observed:** The canonical URL and OG image URL fields accept any text input without format validation. Users can enter "not-a-url" or relative paths that will produce invalid meta tags.
- **Expected:** URL fields should validate format before saving.
- **Impact:** Invalid URLs in meta tags can harm SEO and cause broken social media previews.
- **Suggested fix:** Add URL validation: `if (form.canonical_url && !/^https?:\/\/.+/.test(form.canonical_url)) { toast({ title: "Invalid URL format", variant: "destructive" }); return; }`

### Accessibility

#### [UX-028] ProjectCard uses role="button" for navigation
- **Severity:** Critical
- **App:** portfolio
- **Location:** `artifacts/portfolio/src/components/ProjectCard.tsx:34` — `role="button"` with `onClick={() => setLocation("/projects/${slug}")}`
- **Observed:** The project card has `role="button"` but navigates to a new route. Screen readers announce it as "button" when it should be announced as "link." This is semantically incorrect — navigation should use `role="link"` or an `<a>` element. Additionally, the card has `aria-label="View details for ${project.title}"` which is good, but the role mismatch confuses assistive technology.
- **Expected:** Navigation elements should use `role="link"` or `<a>` elements.
- **Impact:** Screen reader users cannot distinguish between buttons and links, breaking their mental model of the page structure.
- **Suggested fix:** Change to `role="link" aria-label={`View details for ${project.title}`}` or wrap the card content in `<Link href={`/projects/${slug}`}>`.

#### [UX-029] Sidebar active nav link missing aria-current
- **Severity:** Medium
- **App:** admin
- **Location:** `artifacts/admin/src/components/Sidebar.tsx:149` — `aria-label={label}` on active link
- **Observed:** The active sidebar navigation link uses `aria-label={label}` which duplicates the visible text. It should use `aria-current="page"` to semantically indicate the current page, which is the standard ARIA pattern for active navigation items.
- **Expected:** Active nav links should use `aria-current="page"`.
- **Impact:** Screen reader users cannot determine which page they are currently on from the sidebar navigation.
- **Suggested fix:** Replace `aria-label={label}` with `aria-current={isActive(path) ? "page" : undefined}`.

#### [UX-030] Header theme toggle missing aria-pressed
